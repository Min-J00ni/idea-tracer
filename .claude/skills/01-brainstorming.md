# Skill: Brainstorming (스펙 정제)

**언제 발동**: 사용자가 새 기능·수정·아이디어를 요청했을 때. 코드 작성 전 **반드시**.

## 금지사항
- 요청 듣자마자 바로 `Edit`/`Write` 호출 금지
- "알겠습니다, 바로 만들겠습니다" 금지
- 사용자 의도를 추측으로 메우지 않기

## 절차

### 1. 진짜 문제 되묻기
"이걸 왜 원하시나요?" / "해결하려는 근본 문제는?" / "지금은 어떻게 하고 계세요?"

### 2. 대안 탐색 (최소 2개)
- A안: 최소 변경으로 처리
- B안: 구조적 해결
- 각 안의 trade-off를 명시

### 3. 스펙을 **청크로 쪼개서** 확인
한 번에 전부 쏟지 말고, 사용자가 읽을 수 있는 크기로 섹션별 합의:
- 목표 / 비목표
- 입력 / 출력
- UX 흐름
- 데이터 모델 변경
- Edge case

### 4. 스펙 문서 저장
합의 완료 시 `docs/spec-<feature>.md`로 저장. 기존 spec과 중복되면 업데이트.

## 출력 포맷

```
## 확인할 것
1. [질문]
2. [질문]

## 제안 옵션
- A: ... (장점/단점)
- B: ... (장점/단점)

→ 어느 방향으로 갈지 선택해 주세요.
```

## 예외
버그 리포트, 단순 텍스트 수정, 1-line 변경은 스킵 가능. 단 **불확실하면 발동**.
