# Skill: Verification Before Completion

**언제 발동**: "완료했습니다" / "동작합니다" / "고쳤습니다"를 **말하기 직전**.

## 원칙: Evidence over claims
타입체크 통과 ≠ 기능 동작. 컴파일 성공 ≠ UX 정상.

## 완료 선언 전 체크리스트

### 코드 변경 공통
- [ ] `pnpm lint` 통과
- [ ] `pnpm tsc --noEmit` 통과 (타입 에러 0)
- [ ] 관련 테스트 실행 → 초록
- [ ] 변경하지 않은 영역도 회귀 없는지 확인

### UI/프론트엔드 변경
- [ ] `pnpm dev` 실행 → 브라우저에서 **실제로** 해당 화면 열기
- [ ] 골든 패스 1회, edge case 1회 이상 수동 조작
- [ ] 콘솔에 새 warning/error 없음
- [ ] 반응형: 모바일 폭에서도 확인 (프로젝트 규칙)

### API 라우트 변경 (stt/analyze/export-notion)
- [ ] 실제 요청 1회 날려 응답 확인 (mock-data로만 검증 금지)
- [ ] 에러 케이스(잘못된 입력, 빈 응답)도 확인
- [ ] `.env.local` 키 노출 여부 재확인

### "테스트할 수 없었다"면
솔직하게 **말한다**. "동작합니다"라고 말하지 않는다.
예: "타입 체크는 통과했지만 STT는 실제 오디오로 검증하지 못했습니다."

## 보고 포맷
```
## 완료 내역
- 변경: <요약>
- 검증:
  - ✅ lint / tsc 통과
  - ✅ <테스트 이름> 통과
  - ✅ 브라우저에서 <구체 시나리오> 확인
- 미검증: <있다면 명시>
```

## 안티패턴 (사용자 신뢰 파괴)
- "잘 될 거예요" — 검증 없음
- "TypeScript 에러 없으니 완료" — 런타임 확인 없음
- "이론상 동작합니다" — 실행하지 않았다는 실토
